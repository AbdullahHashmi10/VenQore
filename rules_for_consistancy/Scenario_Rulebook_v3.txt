S-001  Mixed-Batch Sale — The Core FIFO Example (User's Scenario)
✅  SOLVED
TRIGGER
You buy stock at different times and costs. A sale pulls from multiple batches at once.
NUMBERS
Batch A: 10 units bought @ Rs.50 cost  (sale price Rs.100)
Batch B: 10 units bought @ Rs.100 cost (sale price Rs.200)
You sell 10 units AFTER buying Batch B.
Stock available: 2 units from Batch A + 8 units from Batch B = 10 units
COGS calculation:
  2 units × Rs.50  = Rs.100   (from Batch A — oldest, used first)
  8 units × Rs.100 = Rs.800   (from Batch B — next oldest)
  Total COGS       = Rs.900
If sale price = Rs.200 per unit → Revenue = Rs.2,000
Gross Profit = Rs.2,000 - Rs.900 = Rs.1,100
PROBLEM
If the system uses one 'average cost' instead of FIFO batches, it would use Rs.75 cost per unit → COGS Rs.750. That is a Rs.150 error. The profit figure is wrong, inventory valuation is wrong, and the error compounds with every subsequent sale.
THE RULE
→  FifoService.deductStock() ALWAYS queries inventory_batches ORDER BY created_at ASC
→  Each batch deduction creates one row in sale_item_batches: {batch_id, qty_taken, unit_cost}
→  COGS = SUM(sale_item_batches.qty_taken × unit_cost) for this sale — never products.cost_price
→  The sale_item_batches table is the audit trail that proves every COGS calculation
JOURNAL
DR  5000  COGS              = Rs.900
CR  1100  Inventory Asset   = Rs.900
sale_item_batches row 1: batch_A_id, qty=2, unit_cost=50, total=100
sale_item_batches row 2: batch_B_id, qty=8, unit_cost=100, total=800
S-002  Return After Mixed-Batch Sale — Which Batch Gets Restored?
✅  SOLVED
TRIGGER
Customer returns some units from a sale that pulled from multiple batches.
NUMBERS
Sale S-001 above: 2 units from Batch A + 8 units from Batch B
Customer returns 3 units.
Question: do the 3 returned units go back to Batch B (newest) or Batch A?
Wrong approach: create a new batch for the returned units
Correct: restore from most-recently-deducted batch first (reverse of the sale)
  3 units → restored to Batch B first (deducted last, restored first)
  If return &gt; 8, overflow goes to Batch A
  Restored cost = the EXACT unit_cost from the original sale_item_batches row
PROBLEM
If a new batch is created for returned items, the cost gets wrong. If average cost is used, existing batch costs are corrupted. The only correct method is to reverse into the exact original batches using the sale_item_batches audit trail.
THE RULE
→  On a sale return: read sale_item_batches for the original sale, ordered by most-recent-deduction first
→  Restore remaining_qty on the ORIGINAL batch_id — not a new batch
→  The COGS reversal = the exact cost from sale_item_batches (not current batch cost)
→  If the original batch has been fully deleted or archived: edge case — see S-009
JOURNAL
DR  1100  Inventory Asset   = Rs.300  (3 units × Rs.100 from Batch B)
CR  5000  COGS              = Rs.300
UPDATE inventory_batches SET remaining_qty = remaining_qty + 3 WHERE id = Batch_B_id
S-003  Cost Price Changes on Next Purchase — Old Stock Not Affected
✅  SOLVED
TRIGGER
Same product bought twice at different prices. Do old unsold units change cost?
NUMBERS
Jan: 50 units @ Rs.80 cost → Batch A  (30 sold, 20 remaining)
Feb: 50 units @ Rs.120 cost → Batch B  (all 50 remaining)
Mar: Sell 30 units
FIFO: deduct from Batch A first (20 remaining), then from Batch B
COGS = 20 × Rs.80 + 10 × Rs.120 = Rs.1,600 + Rs.1,200 = Rs.2,800
The Feb purchase at Rs.120 does NOT change Batch A's cost — it stays at Rs.80 forever
PROBLEM
If the system updates products.cost_price when a new purchase arrives and uses that for COGS, all remaining old stock gets repriced. The Jan stock that cost Rs.80 would be reported as costing Rs.120. COGS is wrong, profit is wrong, and the error is invisible.
THE RULE
→  inventory_batches.unit_cost is LOCKED at the moment of purchase — NEVER overwritten
→  products.cost_price is a display field only — never used for any COGS calculation
→  Each batch is independent — a new purchase creates a new batch, leaves all prior batches untouched
S-004  Zero-Cost Items — Free Samples or Gifts Received
✅  DECIDED — Warn but allow
TRIGGER
Stock received for free — promotional samples, gifts from supplier, items made from scraps.
NUMBERS
Supplier sends 10 free sample units alongside a paid purchase
Or: 5 units assembled from scrap material at zero incremental cost
If unit_cost = 0:
  COGS when sold = Rs.0
  Inventory value contribution = Rs.0
  Gross Profit = 100% of sale price
PROBLEM
Allowing zero-cost batches means items appear to generate infinite margin. If this is not intentional, it may hide real cost. However, zero-cost stock is valid in some businesses.
THE RULE
→  DECISION: zero-cost batches ARE permitted — business chooses the cost at time of entry
→  System must WARN (not block) when a purchase line has unit_cost = 0
→  The warning must be acknowledged — prevents accidental zero-cost entries
→  Zero-cost items sold: COGS = Rs.0, Revenue = sale price, Gross Profit = Revenue (valid)
→  Separate line on purchase: qty_free is recorded at Rs.0 with a 'promotional' flag
⚠️ WATCH
If someone enters a Rs.10,000 item at zero cost by accident, profit is overstated. The warning step is mandatory.
S-005  Oversell — Sale Quantity Exceeds Available Stock
✅  SOLVED
TRIGGER
User tries to sell 20 units but only 15 are in stock.
NUMBERS
Stock: 15 units across 2 batches
Sale attempted: 20 units
FIFO deduction: Batch A: 10 taken (now 0 remaining), Batch B: 5 taken (now 0 remaining)
Remaining needed: 5 units — no batches left
FifoService raises InsufficientStockException
PROBLEM
If the system allows the sale to proceed, remaining_qty on batches goes negative. Inventory value goes negative. COGS is partially calculated. The journal entry is posted but the physical reality doesn't match.
THE RULE
→  FifoService throws InsufficientStockException when remaining_needed &gt; 0 after exhausting all batches
→  The entire transaction is rolled back — DB::transaction() ensures nothing is saved
→  User sees: 'Insufficient stock — 15 units available, 20 requested'
→  EXCEPTION: if the business allows backorders, a flag can be set per product. In that case, a backorder record is created but the stock journal entry is held until restocking
→  Default behaviour: BLOCK the sale — no negative stock permitted
S-006  Multi-Warehouse FIFO — Which Warehouse Does Stock Come From?
✅  DECIDED — Per-warehouse FIFO (ORDER BY created_at ASC within warehouse)
TRIGGER
Business has 2+ warehouses. A sale is made — which warehouse's stock is deducted?
NUMBERS
Warehouse A: 10 units (bought Jan @ Rs.50)
Warehouse B: 5 units  (bought Mar @ Rs.80)
Sale from Warehouse B: 3 units
Option 1 — Per-warehouse FIFO: oldest stock IN THAT WAREHOUSE deducted first
Option 2 — Global FIFO: oldest stock in ANY warehouse deducted, regardless of location
PROBLEM
If FIFO is global but the goods are physically in a different warehouse, stock transfer (B11) is needed before sale. If FIFO is per-warehouse, batches in Warehouse B may be newer and more expensive than global oldest. The choice affects COGS.
THE RULE
→  DECISION TAKEN: FIFO is PER-WAREHOUSE
→  Rationale: you can only sell what is physically at the point of sale location
→  FifoService.deductStock() filters by warehouse_id FIRST, then orders by created_at ASC
→  If a sale specifies Warehouse B: oldest batch IN Warehouse B is used first
→  Stock transfers (B11) should happen BEFORE a sale if goods need to come from another location
⚠️ WATCH
Document clearly in UI: when creating a sale, the warehouse selection determines which stock is deducted.
S-007  Expired Batch — Goods Past Expiry Still in System
⚠️  PARTIALLY HANDLED
TRIGGER
Batch with an expiry date that has passed is still showing as available stock.
NUMBERS
Batch C: 20 units, expiry = 2025-12-31, today = 2026-03-05
System shows: 20 units available
FifoService would deduct from this batch first (it's oldest)
Selling expired goods = legal/quality risk
PROBLEM
The system has no expiry enforcement. It will happily sell expired stock as the first batch in FIFO order. The D23 Expiry Report exists but no system block prevents the sale.
THE RULE
→  RULE: inventory_batches.expiry_date IS stored (nullable). If populated and date &lt; today: batch is BLOCKED from FIFO deduction
→  FifoService adds: AND (expiry_date IS NULL OR expiry_date &gt;= CURDATE()) to batch query
→  Blocked-expiry stock must be written off via B10 Stock Adjustment (Loss) to remove from books
→  D23 Expiry Report shows batches expiring in next 30/60/90 days as warnings
→  PARTIAL: the report exists. The FifoService filter clause needs to be added — currently missing
S-008  Batch with remaining_qty = 0 — Should It Still Appear?
✅  SOLVED
TRIGGER
A batch has been fully sold or written off. Does it stay in inventory_batches?
NUMBERS
Batch A: 10 units purchased, 10 sold → remaining_qty = 0
Option 1: Delete the row → cheaper queries, lose audit trail
Option 2: Keep row with remaining_qty = 0 → audit intact, query must filter it
PROBLEM
If zero-qty batches are deleted, the sale_item_batches table has orphaned references. Returns become impossible to trace. The audit trail is destroyed.
THE RULE
→  Batches are NEVER deleted — remaining_qty = 0 rows stay forever
→  FifoService always adds WHERE remaining_qty &gt; 0 to the deduction query
→  sale_item_batches references (batch_id) remain valid for all time — return tracing always works
→  Index on (product_id, remaining_qty, created_at) ensures zero-qty rows don't slow queries
S-009  Return After Original Batch Was Fully Sold — Edge Case
✅  SOLVED
TRIGGER
All units from the original batch were sold to different customers. One customer returns. The batch has remaining_qty = 0.
NUMBERS
Batch A: 10 units → sold 5 to Customer X, sold 5 to Customer Y
Batch A remaining_qty = 0
Customer X returns 2 units
The batch still EXISTS with remaining_qty = 0
Returning to it: remaining_qty becomes 2 (valid — the row allows positive)
PROBLEM
If zero-qty batches were deleted, this return would be impossible to process correctly. The returned units would have to create a new batch with potentially wrong cost.
THE RULE
→  Return always goes back to the original batch_id from sale_item_batches
→  remaining_qty can increase from 0 — there is no constraint preventing this
→  The returned unit_cost = the original sale_item_batches.unit_cost (not recalculated)
→  COGS is reversed at the original cost — inventory is restored at the original cost
S-010  Trade Discount on Purchase — Does It Affect Batch Cost?
✅  SOLVED
TRIGGER
Supplier gives 10% trade discount. Do you record the gross cost or the net cost in the batch?
NUMBERS
Invoice: 100 units × Rs.200 = Rs.20,000
Trade discount: 10% = Rs.2,000
Net cost paid: Rs.18,000
unit_cost in batch: should it be Rs.200 or Rs.180?
PROBLEM
If batch stores Rs.200 but you only paid Rs.180, your COGS is overstated and inventory is overvalued. Your gross profit is understated on every unit sold from this batch.
THE RULE
→  Batch unit_cost = net cost paid = Rs.180 (after all trade discounts)
→  Trade discounts reduce the cost of acquisition — they are not separate income
→  Journal entry: DR 1100 Inventory Rs.18,000 / CR Cash Rs.18,000 (net amount only)
→  The Rs.2,000 discount is never posted separately — it simply reduces the purchase value
→  DISTINCTION: cash discount for early payment → that IS posted separately (to Other Income 4100)
S-011  Selling Below Cost — Negative Margin Sale
✅  DECIDED — Warn + Manager PIN required
TRIGGER
Item costs Rs.100. You sell it for Rs.80 (clearance, error, or competitive pressure).
NUMBERS
unit_cost = Rs.100,  sale_price = Rs.80
COGS = Rs.100,  Revenue = Rs.80
Gross Profit on this item = -Rs.20  (a loss)
PROBLEM
Should the system allow this? It is financially valid — businesses sometimes sell below cost. But it may also be a data entry error that should be flagged.
THE RULE
→  DECISION: system WARNS but does not BLOCK when sale_price &lt; unit_cost
→  Warning shown at line item level: 'Selling below cost — margin is negative'
→  A supervisor PIN or acknowledgement is required to proceed
→  The journal entries are exactly the same — COGS still = actual purchase cost
→  The loss is correctly reflected in Gross Profit on the P&amp;L
⚠️ WATCH
Never change the COGS to match the sale price. The true cost must always be recorded even if it creates a negative margin.
S-012  Unit of Measure Mismatch — Bought in KG, Sold in Grams
✅  DECIDED — Conversion support in Phase 1 (B31)
TRIGGER
A product is purchased as 5kg but needs to be sold in 500g portions (10 portions per kg).
NUMBERS
Purchase: 5 kg @ Rs.200/kg → batch cost = Rs.200/kg
Sale unit: 500g = Rs.110 per portion
If system records qty=5 (kg) and you sell qty=1 (portion):
  FIFO deduction: 1 unit deducted from batch
  COGS = Rs.200 (wrong — 1 portion costs Rs.100, not Rs.200)
PROBLEM
Unit of measure conversion is not defined in the current system. Buying in one unit and selling in another will produce wrong COGS and wrong stock counts unless the conversion factor is handled.
THE RULE
→  DECISION: UOM conversion factor stored on the product: purchase_uom, sale_uom, conversion_factor
→  Example: purchase_uom = kg, sale_uom = 500g, conversion_factor = 2 (2 sale units per purchase unit)
→  When selling 1 portion: FIFO deducts 0.5 kg from the batch
→  unit_cost for COGS = batch.unit_cost ÷ conversion_factor = Rs.200 ÷ 2 = Rs.100 per portion
→  Stock display: shows in sale_uom (portions) unless user switches to purchase_uom (kg)
S-013  Manufacturing Produces Fewer Units Than Planned — Wastage
✅  SOLVED
TRIGGER
BOM says 10 inputs make 100 units. Actual run produces only 92 due to waste.
NUMBERS
Raw material consumed: 10 kg flour @ Rs.50 = Rs.500 total cost
Expected output: 100 biscuit packets
Actual output: 92 biscuit packets (8 lost to breakage/waste)
Cost per unit if spread across 92:  Rs.500 ÷ 92 = Rs.5.43 per packet
Cost per unit if spread across 100: Rs.500 ÷ 100 = Rs.5.00 per packet
PROBLEM
If the system uses the planned quantity (100) for cost calculation, each finished unit is undercosted. Real cost is higher. The waste is invisible — it just reduces the margin silently.
THE RULE
→  Finished goods batch unit_cost = total_production_cost ÷ actual_units_produced (not planned)
→  Wastage is automatically absorbed into the cost of the surviving units
→  If specific waste tracking is needed: create a separate B10 Loss entry for wasted materials before capitalising finished goods
→  Production run records both: units_planned and units_produced — variance is visible in production reports
S-014  Sub-Assembly — Finished Good A Is an Input to Finished Good B
✅  DECIDED — Maximum 5 levels of BOM nesting
TRIGGER
You manufacture Item A. Then Item A is used as a component to manufacture Item B.
NUMBERS
Step 1: Make Item A (sub-assembly) — raw materials → Item A batch, unit_cost = Rs.120
Step 2: Make Item B — uses 2× Item A + other materials
  Item A cost contribution: 2 × Rs.120 = Rs.240
  Other materials: Rs.80
  Total Item B cost: Rs.320
PROBLEM
The BOM for Item B references Item A as a component. When a production run for Item B runs, it needs to deduct from Item A's inventory_batches using FIFO — just like a sale. Then capitalise the total cost (incl. Item A's FIFO cost) into Item B's batch.
THE RULE
→  Item A is a manufactured product with its own inventory_batches
→  When Item B production consumes Item A: FifoService.deductStock(Item_A) is called — same as selling
→  Item B's batch cost = SUM of all consumed items at their respective FIFO costs + labor
→  Circular BOM must be blocked by system (A needs B needs A = infinite loop)
→  BOM levels: support up to 5 levels of sub-assembly (beyond that, most businesses don't need)
S-015  Production Run Reversed — Finished Goods Go Back to Raw Materials
✅  DECIDED — Partial reversal of unsold qty only
TRIGGER
A production run was posted in error. How do you undo it?
NUMBERS
Production Run #42: consumed 10 kg flour (Batch A) → created 100 biscuit packets (Batch F1)
Error discovered: wrong recipe was used
Need to: destroy Batch F1 (finished goods), restore Batch A (raw materials)
Some packets may already have been sold — those can't be restored
PROBLEM
A straight reversal is only possible if none of the finished goods have been sold. If some are sold, partial reversal is needed. This is a complex multi-step operation.
THE RULE
→  RULE: check sale_item_batches for any deductions from Batch F1 before allowing reversal
→  If zero units sold: full reversal — reverse production journal, delete Batch F1, restore Batch A
→  If some units sold: only unsold units can be reversed — partial reversal
→  Partial reversal: remaining_qty of Batch F1 set to 0 (write-off), Batch A partially restored
→  All reversal entries are posted as new journal entries — never delete the originals
→  A 'production reversal' transaction type (B16-REV) must be tracked separately in the reference table
S-016  Serial Number Appears on Two Batches — Data Entry Error
✅  SOLVED
TRIGGER
Same IMEI/serial number was entered on two separate purchases.
NUMBERS
Purchase Jan: IMEI 123456789 → Batch A, product = Phone X
Purchase Mar: IMEI 123456789 → Batch B, product = Phone X (duplicate entry)
Selling IMEI 123456789 — which batch?
PROBLEM
Serial numbers are unique identifiers — duplicates mean a physical item is tracked twice. Selling it once from each batch doubles the revenue and creates phantom inventory.
THE RULE
→  UNIQUE constraint on (serial_number, product_id) in the serial_numbers table
→  The database rejects the duplicate at INSERT time — the user sees an error before saving
→  The error message: 'Serial number 123456789 already exists on Purchase #[original_id]'
→  Resolution: user must investigate — was it a duplicate entry or a genuinely different unit?

S-017  Customer Overpays an Invoice
✅  SOLVED
TRIGGER
Invoice total Rs.10,000. Customer pays Rs.11,000. Rs.1,000 excess.
NUMBERS
Invoice #501: Rs.10,000
Payment received: Rs.11,000
After allocating Rs.10,000 to Invoice #501: Rs.1,000 unallocated
payment_allocations: {journal_entry_id: PAY1, sale_id: 501, allocated_amount: 10000}
Unapplied credit: {journal_entry_id: PAY1, sale_id: NULL, allocated_amount: 1000}
PROBLEM
The full Rs.11,000 must be recorded as received. The unallocated Rs.1,000 sits as a credit in the customer's account (reduces their Receivable below zero for this amount). It must be either refunded or applied to their next invoice.
THE RULE
→  Post the full Rs.11,000 payment: DR Cash Rs.11,000 / CR AR Rs.11,000
→  payment_allocations: one row for Invoice #501 (Rs.10,000) + one row with sale_id=NULL (Rs.1,000)
→  Customer's AR balance becomes -Rs.1,000 (credit balance — you owe them)
→  Apply to next invoice: create another payment_allocations row when next invoice is issued
→  If refunding: post B4-refund: DR AR Rs.1,000 / CR Cash Rs.1,000
S-018  One Payment Covers Multiple Invoices
✅  SOLVED
TRIGGER
Customer pays Rs.25,000 to settle three outstanding invoices.
NUMBERS
Invoice #601: Rs.8,000   Invoice #602: Rs.12,000   Invoice #603: Rs.5,000
Total owed: Rs.25,000.  Payment: Rs.25,000.
payment_allocations rows:
  {PAY2, sale_id:601, Rs.8,000}
  {PAY2, sale_id:602, Rs.12,000}
  {PAY2, sale_id:603, Rs.5,000}
SUM = Rs.25,000 = payment total  ✓
PROBLEM
Without payment_allocations, all three invoices still show as Unpaid even though the full amount was received. The total AR balance is correct but individual invoice status is broken.
THE RULE
→  One journal entry: DR Cash Rs.25,000 / CR AR Rs.25,000
→  Three payment_allocations rows — one per invoice
→  System must show the user a list of open invoices to allocate against
→  Default allocation order: oldest invoice first (FIFO on invoices) unless user specifies otherwise
S-019  Payment Allocated to Wrong Invoice — How to Correct
🔧  DESIGN DECISION REQUIRED
TRIGGER
Rs.5,000 was allocated to Invoice #700 but should have been Invoice #701.
NUMBERS
Current state:
  payment_allocations: {PAY3, sale_id:700, Rs.5,000}
  Invoice #700 shows Paid. Invoice #701 still shows Unpaid.
Correct state should be:
  Invoice #701 → Paid. Invoice #700 → Unpaid.
PROBLEM
The journal entry (DR Cash / CR AR) is correct — the money was received. Only the allocation is wrong. Deleting the journal entry would be wrong. Only the payment_allocations row needs to change.
THE RULE
→  A 'Reallocate' function: delete the wrong payment_allocations row, insert the correct one
→  The journal entry is NOT touched — it remains as originally posted
→  Total of payment_allocations for this payment must still equal the payment journal entry amount
→  Audit trail: reallocation creates a log entry: {changed_by, changed_at, old_sale_id, new_sale_id}
→  This operation requires supervisor access — it changes invoice badges without posting new journals
S-020  Bounced Cheque — Payment Recorded But Money Never Arrived
✅  SOLVED — B25 B-REV Transaction
TRIGGER
Customer paid by cheque. It was recorded in AMD. The cheque bounced 3 days later.
NUMBERS
Day 1: Cheque received → posted as B4 Payment In
  DR Cash Rs.15,000 / CR AR Rs.15,000
  Invoice #801 shows Paid.
Day 4: Cheque returned by bank — funds never deposited
Now: Cash shows Rs.15,000 more than actual. Invoice shows Paid when it isn't.
PROBLEM
No specific 'bounced cheque' transaction type or reversal path exists in the current system. Law 2 says entries are never deleted, but this needs a clean reversal mechanism.
THE RULE
→  NEW TRANSACTION TYPE: B-REV (Payment Reversal)
→  Step 1: Reverse the original payment entry:
→    DR AR Rs.15,000 / CR Cash Rs.15,000 (exact mirror of original, marked as reversal)
→  Step 2: Set payment_allocations status = REVERSED (never delete — preserve audit trail)
→    Invoice #801 reverts to Unpaid
→  Step 3: Add bank charges if the bank charged a dishonour fee:
→    DR 6000 Operating Expenses / CR Cash (bank charge)
→  The customer still owes Rs.15,000 + any bank charges — their balance is restored
S-021  Bad Debt Write-Off — Customer Will Never Pay
✅  SOLVED — B26 + Account 6700
TRIGGER
Customer owes Rs.30,000. They have disappeared or gone bankrupt. Debt is uncollectable.
NUMBERS
Customer AR balance: Rs.30,000  (3 invoices)
Decision: write off as bad debt
Incorrect approach: delete the invoices (destroys history)
Incorrect approach: post as Fund Out B14 (wrong — this is not an owner drawing)
Correct approach: recognise as an expense
PROBLEM
No bad debt write-off mechanism exists in the current system. The AR balance never clears unless an improper workaround is used.

→  NEW ACCOUNT: 6700 Bad Debt Expense

→  If customer pays later (recovery): DR 1200 AR / CR 6700 Bad Debt Expense (partial reversal), then post normal B4
→  PARTIAL RECOVERY: if they pay Rs.10,000 of the Rs.30,000 → DR Cash Rs.10,000 / CR 4100 Rs.10,000

S-023  Advance Payment — Partial Application to First Invoice, Remainder to Next
✅  SOLVED

S-024  Return After Customer Already Paid — What Happens to the Cash?
✅  SOLVED
TRIGGER
Customer paid Rs.5,000 for Invoice #1001. They return Rs.2,000 worth of goods.
NUMBERS
Invoice #1001: Rs.5,000  (fully paid)
Return: Rs.2,000 goods
Three possible outcomes:
Option A: refund Rs.2,000 cash immediately
Option B: issue credit note — apply to next invoice
Option C: leave as unapplied credit in customer account
PROBLEM
The system must support all three options. The B8 Sale Return currently handles the revenue and COGS reversal but doesn't address what happens to the already-paid cash.
THE RULE
→  All three options use the same B8 journal entries for revenue/COGS reversal

S-108  Products Bought as a Set, Sold Individually
✅  SOLVED — B30 Disassembly BOM
TRIGGER
You buy a printer set (printer + ink + cable) for Rs.5,000. You sell each component separately.
NUMBERS
Purchase: 1 'Printer Set' @ Rs.5,000 → one batch
Sell: Printer separately, Ink separately, Cable separately
How do you split the Rs.5,000 cost across 3 items?
Option A: allocate cost proportionally (e.g. printer 80%, ink 15%, cable 5%)
Option B: create one product 'Printer Set', sell as a set only
Option C: at purchase, split the set into 3 component products immediately
PROBLEM
Selling individual components of a purchased set is a disassembly process. It's similar to manufacturing in reverse — one input, multiple outputs.
THE RULE
→  DECISION: B30 Disassembly BOM. NRV-based cost allocation. CR 1100 set, DR 1100 per component at allocated cost.
→  Step 1: purchase arrives as 'Printer Set' (one batch)
→  Step 2: disassembly run: consume 1 Printer Set (FIFO deduct), produce 3 components
→    Cost split: specified by user at disassembly time (e.g. Printer Rs.4,000, Ink Rs.750, Cable Rs.250)
→    Each component creates its own inventory batch with its allocated unit_cost
→  Step 3: sell each component independently as normal B1 sales
→  The cost allocation must sum to the original purchase cost
