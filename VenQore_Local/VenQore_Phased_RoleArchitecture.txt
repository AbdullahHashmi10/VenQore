1. Industry Knowledge Not In Your Document
1.1  Custom Permission Overrides (Tenant-Level)
Your architecture uses fixed roles — a Manager always has exactly the same permissions everywhere. This is fine for V1 but will cause problems when a real customer says: &quot;I want my Manager to be able to invite staff but not delete products.&quot;
The industry solution is a permission_overrides table alongside the role system:
Schema Stub (add to DB from Day 1 — even if UI comes in V2)
CREATE TABLE tenant_permission_overrides (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  tenant_id BIGINT UNSIGNED NOT NULL,
  user_id BIGINT UNSIGNED NOT NULL,
  permission VARCHAR(100) NOT NULL,  -- e.g. &apos;staff.invite&apos;
  granted BOOLEAN NOT NULL,           -- TRUE = grant, FALSE = revoke
  granted_by BIGINT UNSIGNED NOT NULL,
  UNIQUE KEY (tenant_id, user_id, permission)
);
This way, your role system stays clean and your permission check becomes: check role → check overrides → resolve. V1 ignores overrides entirely (no rows exist). V2 exposes the UI. Zero migration cost.
1.2  Invitation Token System
Your document mentions &quot;invite staff&quot; as a permission but doesn&apos;t spec the invitation flow. This is one of the most security-sensitive things in a multi-tenant SaaS. The wrong implementation leaks tenant isolation.
Required tables and flow:
staff_invitations table: id, tenant_id, invited_by, email, role, token (UUID), expires_at, accepted_at, revoked_at
Token is single-use, expires in 48 hours, ties to a specific email address
If a user with that email already exists in the system → link them to the tenant (do NOT create a second users row)
If they don&apos;t exist → account creation flow, then link
Critically: the invitation token must be compared in constant time to prevent timing attacks
Security Rule
Never re-use invitation tokens. Never store them in plain text — store a hash, send the plain token in the email. A leaked DB does not expose active invitations.
1.3  Plan Limit Enforcement Architecture
Your document mentions plan limits (Starter/Growth/Business) but doesn&apos;t specify how they&apos;re enforced at the application layer. This is commonly done wrong — either too loosely (limits are bypassable) or too aggressively (creates race conditions).
The correct pattern:
Store current usage counts as cached columns on the tenants table: staff_count, product_count, branch_count, etc.
On every create action, check: tenant.staff_count &lt; plan.max_staff, within a DB transaction
Increment the counter atomically with the insert — never count(*) in real time (slow and racy)
When plan changes (upgrade/downgrade), fire a job that recalculates counters from scratch and updates limits
When downgrading: if existing usage exceeds new plan limit, don&apos;t delete data — lock creation until they&apos;re under the limit
1.4  Soft Delete Strategy
Your document doesn&apos;t mention soft deletes. In a financial system like VenQore, hard deleting anything is almost always wrong — you cannot delete a product that appears in 300 historical invoices.
Rule: All financially-referenced records must use soft deletes (deleted_at timestamp). This includes: products, customers, suppliers, staff members, purchase orders. Only configuration records (categories, tax rates) may be hard-deleted if not referenced.
What This Means for Roles
When a Store Owner removes a staff member, the tenant_users row gets deactivated (is_active = false, deactivated_at = now()) — not deleted. Their historical audit log entries remain valid and traceable. The Owner Transfer flow similarly cannot hard-delete the old owner&apos;s tenant_users row — it gets a role change, not a delete.
1.5  Session Management for Shared POS Tablets
Your document specifies a POS PIN for shared tablets but doesn&apos;t address session management. In retail with shared tablets, this is a real operational headache.
Industry pattern for shared-device POS:
Device registers once with a device_token (long-lived, scoped to the store)
Individual staff log in with their PIN (short-lived session, e.g., 8 hours or end of shift)
PIN is hashed (bcrypt) in the DB — never stored plain
Failed PIN attempts are rate-limited per device (5 attempts → 30 second lockout, logged)
&apos;End Session&apos; button logs out the current PIN user but keeps the device authenticated
All transactions created during a PIN session carry the PIN user&apos;s user_id, not the device_token
1.6  Audit Log Design (Missing from Your Schema)
Your document has platform_activity_log for Tier 1 actions but doesn&apos;t specify a store-level audit log for Tier 2. Every financially-relevant action in a store needs its own audit trail — this is a legal requirement in most jurisdictions and a customer trust feature.
Recommended store_activity_log table
CREATE TABLE store_activity_log (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  tenant_id BIGINT UNSIGNED NOT NULL,
  actor_user_id BIGINT UNSIGNED NOT NULL,
  action VARCHAR(100) NOT NULL,       -- e.g. &apos;sale.created&apos;, &apos;product.price_changed&apos;
  subject_type VARCHAR(100) NULL,     -- e.g. &apos;Sale&apos;, &apos;Product&apos;
  subject_id BIGINT UNSIGNED NULL,
  old_values JSON NULL,               -- snapshot before change
  new_values JSON NULL,               -- snapshot after change
  ip_address VARCHAR(45) NULL,
  is_impersonation BOOLEAN DEFAULT FALSE,
  impersonated_by BIGINT UNSIGNED NULL,  -- HQ user who was impersonating
  INDEX idx_tenant (tenant_id),
  INDEX idx_actor (actor_user_id),
  INDEX idx_action (action),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
The old_values/new_values JSON columns are critical — when a Manager changes a product price, you want to know what it was before. This data is irreplaceable if a customer comes back 6 months later asking &quot;who changed this price?&quot;
1.7  Email Notification Architecture (Not Mentioned)
Your document covers support tickets and billing webhooks but doesn&apos;t define the store-level email notification system. Store users need emails for: low stock alerts, payment received, new staff invitation, cash drawer variance, failed login attempts, and more.
Design decision you need to make before V2: Do store owners configure their own email settings (custom SMTP/Postmark sender), or does all email go through VenQore&apos;s Postmark account? Each has billing implications. For V1, use VenQore&apos;s account. For V2+, allow custom senders.
1.8  The &apos;One Person, Two Roles&apos; Ambiguity
Your document acknowledges a person can hold a Tier 1 role AND a Tier 2 role. But it doesn&apos;t specify how this works in practice. When you (God Admin) log in and navigate to your own store at /app/my-store/dashboard — which role takes precedence? What if your store role is &quot;Cashier&quot; but you need to see financial reports?
Industry decision: Tier 1 always wins. If a Tier 1 user navigates to a store URL, they get God Admin / Platform Manager level read access to that store regardless of their tenant_users role. Their Tier 2 role only matters when they&apos;re acting as a regular store user (e.g., you want to test the cashier experience). Specify a UI toggle: [Switch to Store User Mode] that temporarily drops Tier 1 context.
