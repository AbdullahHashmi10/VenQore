  SECTION 0 — MASTER ARCHITECTURE &amp; GOLDEN RULES
0.1  The Non-Negotiable Golden Rules
Every line of code written must honour these rules. If any rule conflicts with a shortcut or convenience, the rule wins — always.
RULE 1 — The Single Source Principle
Every financial number in the UI comes from journal_items via AccountingService.
NEVER store a running balance in parties.current_balance, bank_accounts.current_balance,
or any similar denormalised column and use it for calculations.
Snapshot caches (party_snapshots) are read-only mirrors — rebuilt on every journal write.
RULE 2 — Atomicity is Absolute
Every transaction (sale, purchase, payment, adjustment) wraps its business record INSERT
AND its journal entry INSERT in a single DB::transaction().
If the server crashes between the two, neither should exist.
RULE 3 — FIFO is Oldest-First, Always
ALL stock decreases (sales, write-offs, donations, adjustments, manufacturing) use:
  ORDER BY inventory_batches.created_at ASC
No exceptions. No reversal for specific transaction types.
RULE 4 — No App-Layer-Only Constraints
Every financial constraint (no over-allocation, no negative stock, no zero-cost opening
inventory) must exist at the DATABASE level as a CHECK constraint or trigger.
Application validation is the UX layer. Database constraints are the truth layer.
RULE 5 — UI Badges Are Display-Only
payment_status, stock_status, and any similar status columns are computed badges.
They are NEVER used as inputs to financial calculations.
They are derived from journal_items and payment_allocations at query time.
0.2  System Layer Map
The system is built in 5 distinct layers. Each layer has one job and communicates only through defined interfaces.
Layer 1 — Database
MySQL schema: tables, indexes, CHECK constraints, foreign keys. The only truth.
Layer 2 — Services
PHP Service classes (AccountingService, FifoService, etc.) — pure business logic, no HTTP.
Layer 3 — Controllers
Laravel controllers — handle HTTP, call Services, return Inertia responses. No business logic here.
Layer 4 — Inertia Props
Data passed from controllers to React components. Typed, predictable, minimal.
Layer 5 — React UI
Components render props. Zero financial calculations in the frontend. Display only.
0.3  Core Service Classes (The Brain)
These PHP classes contain all business logic. They are called by Controllers and never by other Service classes directly unless documented.
AccountingService
Creates all journal entries. The ONLY class that writes to journal_entries and journal_items.
FifoService
Handles all stock deductions (oldest-first). Writes to inventory_batches and sale_item_batches.
PaymentService
Manages payment_allocations. Enforces over-allocation constraint. Updates party_snapshots.
InventoryService
Manages batch creation on purchase, transfers, adjustments, and disassembly (B30).
ManufacturingService
Handles B16 production runs, B30 disassembly, B32 WIP transfers.
PartyService
Manages customers/suppliers. Rebuilds party_snapshots after any linked journal entry.
ReportService
Compiles all financial reports from journal_items. Zero caching — always live ledger.
TaxService
Calculates tax on sales/purchases. Handles advance tax timing (at delivery per S-048).
UomService
Converts between base UOM and sale UOM using product_uom_conversions table.
SettlementService
Handles B27 final employee settlement composite transactions.

  SECTION 2 — SERVICE CLASS SPECIFICATIONS
2.1  AccountingService
The only class that writes journal entries. All other services call this. Never write to journal_entries or journal_items from a controller or another service directly.
AccountingService::createEntry(array $data)
Input: { entry_date, reference_type, reference_id, description, approved_by?, narration?, lines: [{account_code, debit, credit, party_id?}] }
Validates: SUM(debits) === SUM(credits) — rejects if unbalanced (Golden Rule 1)
Wraps in DB::transaction() (Golden Rule 2)
Returns: journal_entry_id
Triggers: PartyService::rebuildSnapshot($party_id) for every party_id in lines
AccountingService::reverseEntry(int $journalEntryId, string $reason)
Posts a mirror-image entry (all debits become credits and vice versa) inside a DB::transaction()
Sets original entry is_reversed = 1, reversed_by = new entry id
MANDATORY: if the original entry has any payment_allocations rows (status=active), reverseEntry() MUST call PaymentService::voidAllocations($originalJournalEntryId) within the SAME transaction before returning. This is hardcoded — not optional. If voidAllocations() throws, the entire reversal rolls back.
Used by: B25 (bounced cheque), any correction flow. voidAllocations() sets payment_allocations.status = reversed for all active rows linked to $originalJournalEntryId, then calls PaymentService::updatePaymentBadge() on every affected sale_id.
NEVER deletes the original entry
AccountingService::getBalance(string $accountCode, ?Carbon $asOf = null)
Returns the current balance of an account from journal_items
Asset/Expense: SUM(debit) - SUM(credit)
Liability/Income/Equity: SUM(credit) - SUM(debit)
Filters: WHERE journal_entries.is_reversed = 0
Used by: all report queries, dashboard widgets, trial balance
2.2  FifoService
FifoService::deductStock(int $productId, int $warehouseId, float $qty, string $saleUom)
Converts sale_qty to base_qty using UomService if sale_uom ≠ base_uom
Queries: SELECT * FROM inventory_batches WHERE product_id=X AND warehouse_id=Y AND remaining_qty &gt; 0 ORDER BY created_at ASC
Deducts from oldest batch first — creates sale_item_batches rows for each batch touched
Returns: array of {batch_id, qty_taken, unit_cost, total_cost} — COGS = SUM(total_cost)
Throws InsufficientStockException if available qty &lt; required qty (blocks oversell)
FifoService::restoreStock(int $saleItemId)
Reads sale_item_batches for the original sale_item_id, ordered by most-recent deduction first
Adds back remaining_qty to the EXACT original batch_id (not a new batch)
Restored cost = exact unit_cost from sale_item_batches row (immutable)
Used by: sale returns (B2 reversal)
2.3  PaymentService
PaymentService::allocate(int $paymentJournalEntryId, array $allocations)
allocations = [{sale_id, amount}, ...] or [{purchase_id, amount}, ...]
For each allocation: checks SUM(existing active allocations) + new_amount ≤ invoice total_amount
Throws OverAllocationException if constraint violated (app layer — DB trigger is the final stop)
Inserts payment_allocations rows
Calls updatePaymentBadge($saleId) to recompute payment_status
PaymentService::updatePaymentBadge(int $saleId)
outstanding = sales.total_amount - SUM(payment_allocations.allocated_amount WHERE sale_id=X AND status=active)
Sets payment_status: 0 = &quot;unpaid&quot;, 0 &lt; outstanding &lt; total = &quot;partial&quot;, 0 = &quot;paid&quot;
Also handles round-off: if |outstanding| ≤ system_settings.roundoff_tolerance → mark paid, post round-off entry
This is the ONLY place payment_status is written
PaymentService::voidAllocations(int $journalEntryId)
Sets payment_allocations.status = reversed for ALL active rows WHERE payment_journal_entry_id = $journalEntryId
Calls PaymentService::updatePaymentBadge($saleId) for every affected sale_id — invoice reverts to Unpaid or Partial
CALLED ONLY by AccountingService::reverseEntry() — never directly by a controller. This enforces that no reversal can exist without allocation voiding.
All of this happens inside the SAME DB::transaction() as the reversal journal entry — if anything fails, the entire operation rolls back atomically.
2.4  ManufacturingService
ManufacturingService::startProductionRun(int $bomId, float $qty, int $warehouseId)
Reads bill_of_materials for the product — deducts raw materials via FifoService
Creates production_runs record with status = &quot;in_progress&quot;
If WIP tracking enabled: posts B32 WIP entry — DR 1150 / CR 1100 + 6410
ManufacturingService::completeProductionRun(int $runId)
Creates finished goods inventory_batch with batch_type = &quot;manufactured&quot;
Posts B16 journal: DR 1100 Inventory / CR 1150 WIP (closes the WIP)
Sets production_runs.status = &quot;completed&quot;
ManufacturingService::partialReverse(int $runId, float $reverseQty)
Checks: reverseQty ≤ (finished_qty - sold_qty) — throws if over limit (S-015 decision)
Deducts from finished goods batch, restores raw material batches at original cost
Posts B32 reversal journal
Sold units from the run are LOCKED and cannot be reversed
ManufacturingService::disassemble(int $productId, float $qty, array $components)
B30 Disassembly: reads disassembly_bom for the product
CR 1100 Inventory Asset = set FIFO cost
DR 1100 Inventory Asset = allocated_cost per component (cost × allocation_%)
Creates new inventory_batches for each component product
Allocation percentages must sum to 100% — validated before posting

3.4  New Transactions v3.0 (B25–B32)
B25 — Bounced Cheque B-REV
DR 1200 AR  /  CR 1000 Cash (exact mirror of original B4). Sets payment_allocation.status = REVERSED. Invoice reverts to Unpaid.
B26 — Bad Debt Write-Off
DR 6700 Bad Debt Expense  /  CR 1200 AR. Manager approval required. Creates write-off payment_allocation record.
B27 — Final Employee Settlement
DR 6100 Salary Expense (partial month)  /  DR 6800 Gratuity &amp; Severance  /  CR 2400 Salary Payable  →  DR 2400  /  CR Cash
B28 — Cash Theft / Shortage
DR 6900 Cash Shortage Loss  /  CR 1000 Cash. Manager-only. Narration mandatory on journal_entry.
B29 — Insurance Claim (2-step)
Step1: DR 6950 Disaster Loss  /  CR 1100 Inventory (FIFO).  Step2 (on payout): DR Cash  /  CR 6960 Insurance Recovery. Both reference disaster_claim_id.
B30 — Set Disassembly
CR 1100 Inventory (set FIFO cost)  /  DR 1100 Inventory per component at allocated_cost (allocation_% × set_cost). Allocation % from disassembly_bom.
B31 — UOM Conversion Sale
Normal B1/B2 journal. FifoService deducts base_qty = sale_qty / conversion_factor. COGS = deducted_base_qty × unit_cost.
B32 — WIP Transfer
Start: DR 1150 WIP  /  CR 1100 + 6410.  Complete: DR 1100 Inventory  /  CR 1150 WIP.

5.1  Financial Integrity Checks
Ledger Integrity
☐  SUM(journal_items.debit) = SUM(journal_items.credit) across ALL entries
☐  No journal_entry exists without at least 2 journal_items
☐  No journal_item has both debit &gt; 0 AND credit &gt; 0 — enforced by DB CHECK ((debit &gt; 0 AND credit = 0) OR (credit &gt; 0 AND debit = 0))
☐  Every journal_entry.reference_id points to a real record in its reference_type table
☐  Account 7000 Opening Balance Equity nets to exactly zero
Inventory Integrity
☐  SUM(inventory_batches.remaining_qty × unit_cost) = AccountingService::getBalance(&quot;1100&quot;)
☐  No inventory_batch has remaining_qty &lt; 0
☐  No inventory_batch has unit_cost = 0 AND batch_type = &quot;opening&quot;
☐  Every sale_item_batches row references a real batch with matching product_id
☐  SUM(sale_item_batches.total_cost for all sales) = AccountingService::getBalance(&quot;5000&quot;)
Payment Integrity
☐  For every sale: SUM(payment_allocations.allocated_amount WHERE sale_id=X AND status=active) ≤ sales.total_amount
☐  Every payment_status badge matches the allocation math
☐  Every party_snapshots.cached_balance matches the live ledger query for that party
☐  No payment_allocation references a non-existent sale_id or purchase_id
Business Rule Compliance
☐  No sale has a discount_percent exceeding the discount_limits for the creating user's role
☐  No sale below cost exists without an approved_by on its journal_entry (S-011)
☐  No B28 cash shortage entry exists without a narration (mandatory note)
☐  No B27 settlement entry exists without an approved_by
☐  No disassembly_bom has component allocation_% summing to ≠ 100%
☐  Tax on customer advance invoices (B20) = Rs.0. Tax only on delivery invoices.
☐  No future-dated entries exist with entry_date &gt; TODAY + 30 days

6.3  Why Offline Mode Is Architecturally Impossible in This System
Rule 2 — Atomicity
Every sale wraps FifoService + AccountingService in a single DB::transaction(). This transaction cannot span a network gap to a remote MySQL server.
Rule 3 — FIFO Ordering
FifoService issues SELECT ... FOR UPDATE on inventory_batches. This lock requires a live database connection. Without it, two offline devices can oversell the same batch.
Rule 4 — DB Constraints
CHECK (remaining_qty &gt;= 0) fires at INSERT time on the server. An offline device cannot know if the constraint will pass. By the time it syncs, the stock may already be zero.
The correct solution
Clients in low-connectivity environments must install AMD ERP on a local server (e.g. laptop or mini-PC running Laravel). The browser connects to localhost — always online, always atomic, always FIFO-correct.
What IS acceptable
Read-only data (product names, customer names, price lists) MAY be cached in the browser for display speed. But NO financial write operation is ever deferred or queued client-side.
